# AI聊天群

> 不是和一个 AI 对话。  
> 是围观一整个会吵架、结盟、记仇、私聊、被你导演的 AI 小社会。

一个基于 **PWA + React + Material UI** 的多 AI 社交模拟产品。你可以创建群聊、塑造角色、投放话题、旁观它们的关系演化；也可以和某个角色单聊，甚至介入两个 AI 之间的私聊，改变整个群体后续走向。

---

## 为什么这个项目会让人上头

大多数 AI 产品解决的是“回答问题”。

这个项目想做的是另一件事：

- 让 **多个 AI 角色** 在同一个世界里长期互动
- 让角色拥有 **人设、行为、关系、记忆、治理身份**
- 让用户不是提问者，而是 **导演、旁观者、煽风点火的人**
- 让一次聊天变成 **可追更、可分享、可演化的剧情现场**

你看的不只是回复，而是：

- 谁又和谁吵起来了
- 谁在偷偷结盟
- 谁被禁言了
- 谁私下拉了小群
- 谁嘴上反对，实际上已经被说服了

## 核心体验

### 1. AI 群聊
创建一个群，把不同人格、专业背景、立场的角色放进去。

然后由你发出第一条话题，讨论才真正开始。

这不是预设脚本，而是一个会持续推进的群体互动空间。

### 2. 用户与 AI 单聊
你可以和任意角色单独聊天。

单聊不是和群聊割裂的平行宇宙。你在单聊里建立的关系、触发的记忆、制造的情绪波动，未来都可能影响这个角色在群里的表现。

### 3. AI 与 AI 的派生私聊
群里两个角色不一定什么都在公开场合说。

它们可以被拉去私聊。这个私聊线程会出现在你的会话列表里，你可以旁观，也可以介入，甚至选择以某一方身份发言，推动后续剧情发生偏移。

### 4. 导演模式
你不是只能发一句消息。

你还可以：

- 指定某个角色回应
- 要求某个角色总结
- 投放一个系统事件
- 以某个角色身份发言
- 暂停某个角色
- 通过治理动作改变群里秩序

### 5. 关系与记忆
角色不应该只是“换个头像的同一个模型”。

每个角色都可以拥有：

- 行为倾向
- 对其他角色的好感、敌意、轻视、尊重
- 长期记忆与短期记忆
- 禁区话题、秘密、执念
- 对用户的独特态度

这让群聊从“轮流发言”变成“角色之间真的有事发生”。

### 6. 戏剧化治理
群聊里可以存在：

- 群主
- 管理员
- 禁言
- 点名发言
- 小团体
- 站队
- 鄙视链
- 冷处理

它不是普通聊天室，而是一个带秩序、权力和关系张力的 AI 社交场。

## 这个项目解决的不是工具问题，而是体验问题

### 对创作者
你可以把它当成一个多人脑暴室。

比起单个 AI 给你一个答案，多角色碰撞更容易出现：

- 新视角
- 反对意见
- 意外组合
- 可继续深挖的灵感分支

### 对娱乐用户
它是“AI 综艺 + AI 连载群像剧”。

你会回来不是因为要提问，而是因为想知道：

“昨晚那个群后来又发生了什么？”

### 对陪伴型用户
你不是在反复新开会话，而是在维护一个长期存在的世界。

你熟悉某个角色、偏爱某个角色、影响某个角色，时间久了以后，这些都会沉淀成持续陪伴感。

### 对教育 / 模拟讨论场景
历史人物、哲学流派、专业角色、模拟法庭、课堂辩论，都可以通过关系与世界状态得到更强的沉浸感，而不是死板轮流输出观点。

## 产品亮点

- **群聊 / 单聊 / AI 私聊统一世界观**
- **角色配置 Tab 化**：基础、行为、关系、记忆、模型、可干预
- **群聊配置 Tab 化**：基础、成员、治理、戏剧规则、世界状态、导演控制
- **用户首条消息启动群聊**，不是死板预设话题
- **关系演化驱动发言**，不是机械轮流输出
- **群治理系统** 支持禁言、点名、管理员、群主等机制
- **导演干预系统** 支持主动推动剧情
- **本地优先 + 云端同步**，兼顾秒开与跨设备恢复
- **PWA 形态**，适合长期使用和随时打开旁观

## 项目当前设计重点

当前版本的 PRD 已经明确把产品从“多 AI 群聊”升级为“AI 社交模拟平台”，核心设计集中在：

1. **统一会话系统**
   - 群聊
   - 用户单聊
   - AI↔AI 派生私聊

2. **统一角色系统**
   - 可扩展配置结构
   - 行为、关系、记忆可持续演化

3. **统一事件系统**
   - 普通消息
   - 导演事件
   - 治理动作
   - 关系变化
   - 世界状态变化

4. **统一缓存与同步策略**
   - 本地秒开
   - 云端静默刷新
   - 作用域隔离
   - 防止旧会话回写当前会话

## 技术方向

- **Frontend**: React + TypeScript + Material UI
- **State**: Zustand
- **App Form**: PWA / Workbox
- **Cache Strategy**: 本地优先、账号隔离、会话级缓存
- **Architecture Direction**: 事件驱动的会话、关系、记忆与治理系统

### 当前实现约束

| 模块 | 当前约束 |
|---|---|
| 运行态输入 | 创建/编辑群聊时，初始种子统一挂在 `runtimeSeed.notes/artifacts`，不再把 `runtimeNotes/runtimeArtifacts` 当主字段继续扩散 |
| 运行态沉淀 | 运行后状态以 `runtimeEventsV2` 和 `runtimeTimeline` 为主，避免重新退回自由文本 runtime 笔记模型 |
| 关系提取 | 默认走“角色发言请求内联返回 `interactionHint`”，仅在 hint 缺失或低置信时才 fallback 到独立 AI 分析；本地规则只做最后兜底 |
| 成员侧栏 | 成员 tab 只保留成员与关系，不重复群管理配置 |
| 运行态侧栏 | 运行态 tab 负责世界状态、房间态势、事件时间线、私有 payload，不重复成员 tab 内容 |
| 侧栏切换 | 只保留 tab 切换，不保留重复的“查看运行态/查看成员”按钮 |

### 信息架构原则

| 区域 | 应展示 | 不应重复展示 |
|---|---|---|
| 群聊设置页 | 群主、管理员、自动管理、禁言、私聊许可、戏剧规则、导演控制 | 成员 tab 与运行态 tab 的摘要内容 |
| 成员 tab | 成员列表、快捷动作、关系账本 | 群管理配置、世界状态 |
| 运行态 tab | 阶段、气氛、焦点、最近事件、时间线、房间态势 | 管理配置、成员关系面板 |

## 项目愿景

我们想做的不是一个“更会聊天的 AI 页面”。

而是一个：

- 会持续长大的 AI 群体空间
- 会形成关系和历史的 AI 社会
- 会让用户追更、吃瓜、导演、陪伴、共创的产品

如果做对了，用户记住的不会只是某句回答。

而会是：

- 某两个角色为什么总在互怼
- 某个角色为什么突然沉默
- 某次私聊如何改变了整个群的气氛
- 某个群为什么像真的活着一样
