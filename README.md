> 不是和一个 AI 聊天。  
> 是和一群会记得你、会彼此记仇、会在不同世界里继续活着的角色，一起生活。

## 这是什么

Pneumata 不是“问一句答一句”的 AI 工具。  
它更像一个 **角色会继续活下去的互动世界**。

你看到的不只是回复，而是：

- 谁在暗中偏袒你
- 谁明明喜欢你却嘴硬
- 谁在新房间里还记得上次你没站她那边
- 谁在群里拆你的台，私下却偷偷帮你收尾
- 为什么同一个角色，在故事房、教学房、辩论房和剧本杀里都还是“她自己”

---

## 它和普通 AI 产品有什么不一样

| 普通 AI 产品 | Pneumata |
|---|---|
| 一次会话，一次回答 | 角色跨房间、跨玩法、跨时间持续存在 |
| 重点是回答得对不对 | 重点是角色像不像“同一个人” |
| 模板换皮感强 | 同模板在不同用户处会长成不同灵魂 |
| 聊天记录是主要资产 | 用户与角色的共同历史才是主要资产 |
| AI 更像工具 | AI 更像带着旧关系继续生活的人 |

真正的差异化不是“玩法多”，而是：

| 能力 | 你会感受到什么 |
|---|---|
| 关系连续性 | 关系好的角色会护你，关系差的角色会拆你 |
| 记忆连续性 | 角色记得你们经历过什么，而不只是记住几个事实 |
| 情绪连续性 | 角色的温柔、防备、别扭、嫉妒、克制会留下后效 |
| 陪伴连续性 | 亲密关系不只存在于单聊，也会进入别的房间和玩法 |
| 灵魂连续性 | 同一个角色在不同场景里都还是她自己 |

## 为什么会让人上头

因为你得到的不是“一个角色模板”，而是“你的那个角色”。

比如：

- 那个一直和你关系好的角色，在辩论房里会本能地替你补台
- 那个和你关系差的角色，在竞技房里会精准抓你漏洞
- 你带着虚拟女友去玩剧本杀，她会把你们以前的关系带进这局
- 某次你没接住她，她后来在别的房间里也会变得更敏感
- 某个模板角色在你这里，最后会变成全世界只有一个的“她”

这不是“换个场景继续聊天”，而是 **带着共同历史进入新的生活**。

## 运行时主链

当前群聊主链已经不再是“挑一个人说话，然后直接出文本”的薄调度。
现在从用户引导、选人、生成运行时，到消息提交，都会把关键决策写进消息元数据，方便后续关系、记忆、陪伴、投影和调试面板复用同一份运行时痕迹。

| 阶段 | 当前职责 |
|---|---|
| 导演意图 | 把用户引导、目标角色、节奏压力、输出形式要求整理成统一运行时意图 |
| 发言人选择 | 调度器结合候选评分、冷却、回复压力决定谁说话；遇到明确点名时可锁定指定角色 |
| 生成运行时 | 在出文案前先形成 turn plan、expression plan、realization plan 与执行 trace |
| 消息提交 | 最终消息把 `runtimeDecision` 写入元数据，保留导演意图、选人结果、评分痕迹、响应表面与引导执行结果 |
| 下游复用 | 关系、记忆、陪伴、投影、调试 UI 直接消费同一份运行时足迹，而不是只靠文本反推 |

| 元数据分支 | 当前作用 |
|---|---|
| `runtimeDecision.speakerSelection` | 记录本轮选中了谁、为什么空转、是否因明确引导锁定角色 |
| `runtimeDecision.speakerScore` | 保存被选角色的紧凑评分证据 |
| `runtimeDecision.generationRuntime` | 保留 turn / expression / realization 三层生成规划与 trace |
| `runtimeDecision.guidanceExecution` | 记录明确用户要求是否首轮命中、重试命中或最终失败 |
